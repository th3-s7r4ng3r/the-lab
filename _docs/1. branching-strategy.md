# Branching Strategy - The Lab Repository

## Overview

This document outlines the branching strategy adopted for The Lab repository. The strategy is designed to manage IaC changes efficiently across homelab environments efficiently.

## Branch Structure

- **Main Branch**: The `main` branch serves as the production-ready branch for the homelab.
- **Environment Branches**: Separate branches are maintained for each environment:
  - `management`: Managing cluster common resources via IaC
  - `dev-lab`: Development environment for testing homelab tools
  - `main`: Production environment for the homelab
- **Change Branches**: Change branches are created from the respective environment branches for implementing changes.
  - Naming Convention: `env/change-description` (e.g., `dev-lab/add-new-service`, `main/update-configuration`)
- **Hotfix Branches**: Branches created from the `main` branch to address critical issues in production.
  - Naming Convention: `hotfix/issue-description` (e.g., `hotfix/fix-production-bug`)

## Workflow

1. **Creating Change Branches**: Create a change branch from the respective environment branch.
2. **Implementing Changes**: Implement and test changes using Terraform CLI.
3. **Merging Changes**: Once satisfied, merge the change branch directly into the respective environment branch.
4. **Deployment**: Approve the Terraform plan for the respective environment in Jenkins.
5. **Deleting Change Branches**: After merging, delete the change branches to keep the repository clean.

## Special Scenarios

- **Hotfixes**: In case of critical issues in production, hotfix branches (e.g., `hotfix/issue-description`) can be created from the `main` branch, changes are made, and then merged back into `main`. Delete the hotfix branch after merging.
- **Config Changes**: Configuration changes to an environment can be made using a change branch directly from the respective environment branch.
- `cluster-templates` and `jenkins-jobs` directory changes follow the same workflow as above.
- **dev-lab**: This branch is purely for experimenting and testing homelab tools. Changes here are not promoted to `main` unless intentionally adopted.
